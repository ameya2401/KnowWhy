# Purpose

This document defines exactly who KnowWhy is being built for.

One of the most common mistakes made by startups is attempting to solve everyone's problems.

When software attempts to serve everyone, it usually ends up solving nobody's problem particularly well.

KnowWhy deliberately rejects that approach.

Instead of asking:

> "Who could use KnowWhy?"

This document asks:

> "Who experiences the greatest pain that KnowWhy can realistically solve?"

Every future product decision should be validated against these personas.

---

# Design Principle

A product should not have millions of users.

A product should have thousands of users who absolutely love it.

Depth before breadth.

# Primary Target User

Version 1 will primarily target:

Small software teams (5–50 people).

Why?

Because these teams:

Already use many SaaS tools.

Frequently lose context.

Need documentation.

Need workflow assistance.

Generate large amounts of technical knowledge.

Can provide rapid feedback.

Can self-host software if necessary.

Require little enterprise bureaucracy.

---

# Secondary Target User

Growing startups.

Characteristics:

10–100 employees.

Rapid growth.

Knowledge fragmentation.

Frequent onboarding.

Many recurring decisions.

Multiple software platforms.

These companies begin experiencing organizational memory problems before they can afford enterprise knowledge management solutions.

# Why Not Build For Everyone?

Building for everyone means:

Different integrations.

Different workflows.

Different terminology.

Different priorities.

Different UX.

Different pricing.

Different architecture.

This dramatically increases complexity.

Instead,

KnowWhy should solve one group's problems exceptionally well before expanding.

---

# Success Definition

Version 1 succeeds if a small software team says:

> "KnowWhy helped us remember something we would otherwise have lost."

Version 2 succeeds if they begin relying on KnowWhy during everyday work.

Version 3 succeeds if KnowWhy becomes part of their normal workflow rather than an additional tool.

# Final Decision

KnowWhy Version 1 will be designed primarily for software engineering teams and technology startups.

This decision is based on:

- High workflow complexity.
- High context fragmentation.
- Existing AI adoption.
- Fast iteration cycles.
- Easier validation.
- Public datasets.
- Lower integration barriers.
- Strong portfolio relevance.

Future versions may expand into additional industries after achieving strong product-market fit within the initial customer segment.

---

# Closing Statement

KnowWhy should never attempt to become a universal productivity platform.

Instead, it should become indispensable to one specific group of users before expanding.

Focus is not a limitation.

Focus is a competitive advantage.
